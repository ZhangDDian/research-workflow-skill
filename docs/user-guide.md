# AI 辅助调研工作流

## 这是什么？

当你用 AI 编程助手（Claude Code）开发软件时，经常会遇到这样的问题：

- "有没有现成的库可以直接用？"
- "这个技术方案靠不靠谱？"
- "别人是怎么解决这个问题的？"

以前的做法是：随手搜一搜，看到第一个结果就用了。结果经常踩坑——选了个没人维护的库、错过了更好的方案、或者重复造了别人已经造好的轮子。

**调研工作流**就是为了解决这个问题：把"随手搜搜"升级成一套可重复、有质量保证的调研流程，由 AI 自动执行。

## 核心理念

**不重复造轮子。**

每次动手写代码之前，先花几分钟让 AI 帮你调研一下：市面上有没有现成方案？别人是怎么做的？有什么坑要避免？

这不是多余的步骤——这是省时间的步骤。

## 工作流全景

整个调研分 8 步，但你不需要记住每一步。AI 会自动判断该走哪条路线，你只需要提出问题。

```
你提出问题
    ↓
[1] 去重：以前调研过这个话题吗？有就复用
    ↓
[2] 分级：这个问题有多复杂？（快查 / 标准 / 深度）
    ↓
[3] 识别场景：你想"找轮子"还是"评估技术"还是"了解最佳实践"？
    ↓
[4] 澄清：有不确定的地方，先问你 2-3 个问题
    ↓
[5] 告知计划：告诉你"我打算这样调研，用这些工具"
    ↓
[6] 执行调研：AI 自主搜索、筛选、对比
    ↓
[7] 存档：调研报告保存下来，下次可以复用
    ↓
[8] 交付：给你结论 + 下一步建议
```

---

## 调研分三个级别

不是所有调研都需要花大力气。根据问题的复杂度，自动选择合适的投入：

| 级别 | 什么时候用 | 怎么执行 | 大概多久 |
|------|-----------|---------|---------|
| **L1 快查** | "这个 API 怎么用？""配置项叫什么？" | AI 直接查，秒回 | < 1 分钟 |
| **L2 标准调研** | "帮我选个 ORM 框架""有没有好用的 PDF 库？" | 启动一个专门的调研代理 | 15-60 分钟 |
| **L3 深度调研** | "我们该用什么架构？""这个全新领域怎么切入？" | 多个调研代理并行工作 | 1-4 小时 |

**判断规则很简单：**
- 你说"查一下""确认一下" → L1
- 涉及"选型""比较""找开源项目" → L2
- 涉及"完全没接触过的领域""重大技术决策" → L3
- 拿不准？默认 L2，过程中发现更复杂可以升级

## 六种调研场景

不同的问题，用不同的调研方法。AI 会根据你的提问自动匹配：

### W1: 方案发现 —— "有没有现成的？"

> 你说：*"有没有好用的 Markdown 编辑器组件？"*

AI 会做什么：
1. **划定范围** — 你需要哪些功能？必须的和可选的分开
2. **三路搜索** — 同时搜 GitHub 开源项目、产品评测网站、社区推荐
3. **筛选过滤** — 去掉不维护的、许可证不合适的、依赖太重的
4. **对比打分** — Top 3-5 候选方案做对比表
5. **给出建议** — 直接用 / 借鉴核心思路 / 仅作参考

### W2: 用户洞察 —— "用户在抱怨什么？"

> 你说：*"看看大家对现有的 Notion API 有什么不满"*

AI 会做什么：
1. **多渠道采集** — Reddit、Hacker News、知乎、App 评论区、竞品的 GitHub Issues
2. **痛点归类** — 按频率和严重度排序
3. **交叉验证** — 同一个痛点在多个渠道都出现才算数
4. **提炼洞察** — "用户需要 [X]，因为 [Y]"

### W3: 技术评估 —— "这个能不能做？"

> 你说：*"评估一下用 WebSocket 还是 SSE 做实时推送"*

AI 会做什么：
1. **明确约束** — 技术栈、性能要求、部署环境
2. **广度扫描** — 搜索可行的技术方案
3. **候选方案** — 筛出 2-4 个值得深入评估的
4. **复用分析** — 每个候选方案的哪些部分可以直接用、哪些要自己写
5. **决策草案** — 列出选项、各自的代价和收益

### W4: 最佳实践 —— "别人怎么做的？"

> 你说：*"大厂的 Design System 是怎么组织的？"*

AI 会做什么：
1. **搜索案例** — 英文（Exa/Tavily）+ 中文（知乎/36kr）+ 学术论文
2. **排序评估** — 按你关心的标准（规模？效果？成本？）排 Top 5
3. **深入拆解** — Top 2-3 案例的具体做法、适用前提
4. **适配建议** — "他们做了 X，我们的约束是 Y，所以建议改为 Z"

### W5: 知识扩展 —— "最近有什么新东西？"

> 你说：*"LLM 工具调用最近有什么新进展？"*

AI 会做什么：
1. **快速扫描** — Hacker News、Reddit、arXiv、掘金/知乎最新内容
2. **精选信息** — 筛选 ≤10 条高价值信息，每条附来源
3. **关联分析** — 这改变了什么之前的假设？打开了什么新可能？

### W6: 文献研究 —— "学术上怎么看？"

> 你说：*"RAG 检索增强生成的学术共识是什么？"*

AI 会做什么：
1. **建立地图** — 用 Semantic Scholar 找关键论文和引用网络
2. **精读论证** — 每个关键发现附论文出处，注意证据质量层级
3. **对抗检验** — 主动搜索反面意见和局限性
4. **综合回答** — 标注确信度：确信 / 可能 / 不确定

## 两个质量维度

每次调研会叠加两个维度的判断：

### 验证深度 —— 结论需要多可靠？

| 级别 | 什么情况 | AI 会怎么做 |
|------|---------|-----------|
| 🔴 战略级 | 架构决策、方向选择 | 至少找 3 个反面观点，多角度验证，每个关键声明标注出处 |
| 🟡 选型级 | 技术选型、工具对比 | 至少找 1 个反面观点，Top 3 结论标注出处 |
| 🟢 信息级 | 查用法、了解现状 | 快速搜一下有没有反面意见就行 |

### 时效性 —— 信息需要多新？

| 级别 | 什么情况 | 搜索范围 |
|------|---------|---------|
| 🔥 前沿 | 刚发布的 API、具体模型版本 | 限 3 个月内 |
| 📅 近期 | 选型、竞品、行业趋势 | 限 1 年内 |
| 📚 长效 | 方法论、架构模式 | 不限时间，优先权威来源 |

## 工具链

AI 在调研时会用到这些工具（按优先级排列）：

| 用途 | 工具 | 说明 |
|------|------|------|
| 开源项目搜索 | GitHub CLI (`gh`) | 搜索仓库、查看项目详情、Stars、更新时间 |
| 官方文档查询 | Context7 → GitMCP → WebFetch | 从最精准到最通用，逐级降级 |
| 语义搜索 | Exa > Tavily > WebSearch | Exa 理解意图最好，Tavily 实时性强，WebSearch 兜底 |
| 学术搜索 | arXiv / Semantic Scholar | 论文和学术文献 |
| 中文搜索 | Baidu MCP + 知乎/掘金/V2EX | 中文社区和资讯 |
| 实时热点 | HotNews MCP | 新闻和热门话题 |
| 网页抓取 | camoufox（英文）/ Playwright（中文）| 需要反爬能力时使用 |

### 开源项目怎么筛？

不是 Star 多就好。AI 会按这些标准筛选：

- **Stars > 100** — 基本质量门槛
- **最近 6 个月有更新** — 还在维护
- **MIT/Apache/BSD 许可证** — 商用友好
- **Issues 关闭率 > 50%** — 维护者有响应
- **有 README** — 基本文档齐全
- **依赖数量少** — 轻量优先

筛选完，给出采用建议：
- 🟢 **直接用** — 功能匹配、维护好、可以直接 `npm install`
- 🟡 **借鉴实现** — 核心思路好，但依赖太重或功能只部分匹配，读源码后自己实现

## 调研报告去哪了？

每次调研的结果不是用完就扔，而是存档备查：

| 调研类型 | 存在哪 |
|---------|-------|
| 跟当前项目相关的 | 项目目录下 `docs/research/` |
| 跨项目通用的（比如某个技术的通用评估） | 全局 `~/.claude/memory-bank/research/` |
| 临时数据 | `/tmp/`，会话结束可丢弃 |

文件命名规则：`{主题关键词}-{YYYYMMDD}.md`

每个存储目录都有 `INDEX.md` 索引文件。下次调研同类话题时，AI 会先查索引，避免重复劳动。如果已有报告，会问你要不要**复用**（直接引用）、**刷新**（在旧报告基础上更新）还是**忽略**（全新调研）。

### 发现了好的开源项目？

如果调研中发现值得深入研究的开源项目，AI 会：
1. 克隆到本地 `~/projects/repos/`
2. 启动专门的代理做代码级研究
3. 输出分析报告到 `docs/research/{项目名}-analysis.md`

## 调研完了之后呢？

调研不是目的，行动才是。每次调研完成，AI 会：

1. **给出结论摘要** — 关键发现（最多 10 条）
2. **复用分析表** — 每个方案/项目的采用策略和理由
3. **推荐方案** — 综合评估后的建议
4. **PRD 草案**（L2/L3）— 推荐方案、核心功能、技术依赖、风险项、待澄清事项
5. **下一步建议**：
   - 需要编码 → 提示切换到研发模式
   - 需要决策 → 给出"选项 + 推荐"格式

### 重要原则

- 调研阶段**只做信息收集和分析**，不写实现代码
- 调研**不做最终技术决策**——只提供建议，决策权在你手上
