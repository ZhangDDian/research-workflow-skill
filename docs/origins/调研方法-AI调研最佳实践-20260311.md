# 调研报告: AI 辅助调研最佳实践

**日期**: 2026-03-11
**任务**: 研究如何用 AI Agent 高效调研未知领域的最佳实践

---

## 调研摘要

有效的 AI 辅助调研不是"问 AI 一个问题然后接受答案"，而是一套结构化方法论：先规划、再并行广泛搜索、用来源验证每个关键声明、最后带引用综合输出。STORM、Elicit、OpenAI Deep Research、Gemini Deep Research 这些领先系统都收敛到同一架构。人类使用 AI 最有效的方式是把 AI 当作一个"快速但过度自信的合作研究者"而不是神谕。

---

## 现有代码分析

本次调研不涉及代码库分析，为纯技术/方法论调研。

## 技术方案

### 方案 A: STORM 框架 — 多视角综合

**描述**: Stanford OVAL Lab 的 STORM 系统，通过发现多元视角、模拟带有不同视角的对话、策展信息来生成有结构的研究报告。

**核心流程**:
1. 枚举 3-5 个不同的"视角"或"角色"（利益相关者）
2. 从每个视角出发向"主题专家"提问
3. 基于互联网来源收集信息
4. 整理大纲 → 生成带引用的完整报告

**优点**:
- 强制多元视角，减少确认偏差
- 组织性提升 25%，覆盖广度提升 10%（对比基线）
- 有 Co-STORM 版本支持人机协作

**缺点**:
- 主要面向长篇报告生成，不适合快速调研
- 开源系统需要自己部署

**实现复杂度**: 中（使用 storm-project.stanford.edu 公开版本则低）

---

### 方案 B: Deep Research Agent 架构 — 多步搜索循环

**描述**: OpenAI、Gemini、Perplexity 的 Deep Research 产品共用同一核心架构。

**核心流程**:
1. **规划**: Orchestrator 分解问题为子任务，生成研究计划
2. **并行委派**: 多个子 Agent 并发探索不同角度
3. **工具使用**: 网络搜索 + 浏览器（完整页面）+ 代码执行
4. **综合 + 引用**: 合并发现，Citations Agent 验证每个声明

**规模**: 典型查询 ~80 次搜索，~250k tokens

**优点**:
- 自动化程度高，适合复杂多维度问题
- 并行执行节省时间
- Citations Agent 防止幻觉

**缺点**:
- 黑盒，难以干预中间过程
- 成本高，慢查询需要等待

**实现复杂度**: 低（使用现有产品），高（自建）

---

### 方案 C: 人机协作工作流 — Perplexity + Claude 双工具堆栈

**描述**: 功率用户文档最多的双工具工作流。

**核心模式**:
```
Perplexity (发现) → Claude (分析) → Perplexity (验证)
```

**优点**:
- 互补优势：Perplexity 找信息，Claude 理解信息
- 验证闭环减少幻觉风险
- 灵活可控，人在每个环节都可以干预

**缺点**:
- 需要用户主动协调两个工具
- 每月 $40 成本（两个 Pro 订阅）

**实现复杂度**: 低

---

## 推荐方案

**推荐**: 方案 C（人机协作工作流）+ 吸收方案 A 的"多视角"原则

**理由**:
1. 最灵活，适合各种调研场景，不依赖特定产品
2. 多视角原则（STORM 的核心创新）可以手动执行，不需要系统支持
3. 验证闭环是最重要的质量控制机制，而方案 C 天然支持

## 实施建议

### 关键步骤（实用调研工作流模板）

```
Step 0: 定义范围（5分钟）
- 写出需要回答的 3 个具体问题
- 定义"完成"是什么样的
- 估算时间预算

Step 1: 定向（10-15分钟，广度优先）
- 运行 3-5 个广泛查询，使用不同术语
- 扫描结果：识别关键词、关键人物、关键争论
- 不要深读，先建地图
- 输出：词汇表、关键参与者、3-5 个重要子领域

Step 2: 规划（5分钟）
- 选出 2-3 个子领域深入
- 草拟 5-10 个定向查询
- 标注哪些声明需要验证

Step 3: 深度搜索（30-40分钟，深度优先）
- 执行定向查询
- 读原始来源，不只读摘要
- 立即做"声明-来源"配对（声明 + URL）
- 每 5-7 篇来源写一次 3 句话综合

Step 4: 对抗性检验（10分钟）
- 明确搜索反驳和批评
- 问 AI："这个结论有什么问题？"
- 找至少 1 个挑战你结论的来源

Step 5: 综合（15-20分钟）
- 回答 Step 0 的 3 个问题
- 注明仍不确定或未解决的内容
- 标记仍需验证的声明

Step 6: 验证（10分钟）
- 识别 3-5 个最关键声明
- 找每个声明的原始来源
- 核对统计数据与原始研究
```

### 广度优先 vs 深度优先决策

| 情况 | 推荐策略 |
|------|----------|
| 完全未知领域 | 先广度扫描，再选 2-3 个子领域深入 |
| 已知领域，具体问题 | 直接深度优先 |
| 寻找现有解决方案 | 广度扫描（解决方案可能在相邻领域） |
| 理解特定机制 | 深度优先（需要追踪因果链） |
| 时间紧张 | 仅广度优先（获取 70% 的洞察） |
| 高风险决策 | 广度 + 深度 + 对抗性质疑 |

### 风险点

- **确认偏差** - 缓解措施: 明确告知 AI 你的当前假设，要求它找反驳证据
- **幻觉引用** - 缓解措施: 声明-来源配对，关键声明必须核对原始来源
- **覆盖不足** - 缓解措施: Elicit 即使 AI 工具，灵敏度也只有 39.5%，需要多查询策略
- **兔子洞陷阱** - 缓解措施: 预先定义"完成"标准，定期检查是否偏离

### 关键反模式（要避免的）

1. **神谕模式**: 接受 AI 回答而不验证（引用幻觉率可达 86%）
2. **单查询调研**: 只运行 1 个搜索查询
3. **确认螺旋**: 用 AI 寻找支持已有结论的证据
4. **只积累不综合**: 读很多来源但不整合（工作记忆溢出）
5. **信任近期偏见**: 认为最新的 = 最好的
6. **无计划无范围**: 没有定义完成标准就开始调研

### 依赖项

- Perplexity Pro（发现和验证）
- Claude Pro（分析和综合）
- 或 Gemini Deep Research / OpenAI Deep Research（一体化但黑盒）
