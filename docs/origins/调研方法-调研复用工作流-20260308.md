## 发现

### 1. 核心原则

所有项目开始前必须先调研，核心目标：**不重复造轮子**。

调研范围：现有技术方案、社区方案、GitHub/HuggingFace 开源项目、竞品、流行解决方案。

- 不调研就编码 → 禁止
- 调研完必须输出复用分析表格 → 强制

### 2. 调研触发与豁免

**自动进入 RESEARCH 模式的条件：**

| 条件 | 示例 |
|------|------|
| 新功能需求 | "实现"、"添加"、"创建" + 功能描述 |
| 复杂 bug | 非显而易见的问题 |
| 重构请求 | 涉及多文件修改 |
| 不熟悉的代码区域 | 首次接触的模块 |
| 技术选型决策 | 选 A 还是 B |

**跳过调研的条件：**

| 条件 | 示例 |
|------|------|
| 用户明确要求 | "直接改"、"简单修复"、"快速" |
| 显然简单的任务 | 改配置、修 typo、调样式 |
| 用户已提供完整方案 | 附带代码或详细步骤 |
| 极小改动 | 单文件 10 行以内 |

### 3. 调研流程

由 `research-first` Skill 驱动，核心步骤：

1. **复杂度判断**
   - 简单（单文件、确认用法、≤3 次工具调用）→ 主会话内完成
   - 复杂（5+ 文件、网络调研、GitHub 搜索、多模块交互）→ 启动 Researcher Sub Agent

2. **GitHub 搜索**
   - 首选 `gh` CLI（gh search repos, gh repo view, gh api search/repositories）
   - WebSearch 仅作为补充
   - 由 `github-search` Skill 驱动

3. **高价值项目深度研究**
   - 克隆到 `~/projects/repos/{项目名}/`
   - 更新 `~/projects/repos/index.md`（URL、Stars、摘要、调研来源、日期）
   - 启动子代理做代码级研究

4. **输出**
   - 报告保存到 `docs/research/`
   - 必须包含复用分析表格（见下节）

### 4. 三级复用决策框架

每份调研报告的核心产出：

| 策略 | 适用条件 | 行动 |
|------|---------|------|
| **直接使用** | 功能完全匹配、维护良好、依赖可接受 | `npm install` / `pip install` 并集成 |
| **借鉴实现** | 功能部分匹配、依赖过重、需要定制 | 读源码理解核心逻辑，自己实现 |
| **自建** | 无现成方案、差异超过 50%、全新领域 | 从零实现 |

**复用分析表格模板**（调研报告必须包含）：

```markdown
| 方案/项目 | Stars | 策略 | 理由 |
|-----------|-------|------|------|
| {名称}    | {N}   | 直接复用 / 借鉴思路 / 自建 | {简述} |
```

**借鉴实现的具体流程：**

1. 挑选参考项目（代码质量 > Stars）
2. 阅读核心实现文件，理解关键逻辑
3. 记录值得借鉴的设计模式和算法
4. 基于理解写自己的代码（不复制粘贴）

### 5. 经验复用机制

| 机制 | 说明 |
|------|------|
| 启动时扫描 | 项目开始写代码前，读 `~/.claude/memory-bank/learnings/INDEX.md` |
| 调研报告永久保留 | `research/github/`、`research/codebase/`、`research/best-practices/`、`research/competitors/` |
| ADR 防重复否决 | 技术决策前先读 `docs/specs/decisions/`，避免重提已否决方案 |
| 跨项目引用 | 绝对路径（`~/projects/xxx`）；接口文档由提供方维护；依赖反馈记在 `docs/design/dependencies/` |

## 实战案例

### Elvis Agent Swarm — 最完整的复用决策

来源：[RESEARCH-SYNTHESIS.md](../elvis-codex-claudecode/docs/research/RESEARCH-SYNTHESIS.md)

系统分解为 5 个核心模块，已有工具覆盖约 90%，仅编排器无现成方案：

| 模块 | 决策 | 借鉴来源 | 理由 |
|------|------|---------|------|
| git worktree 管理 | 借鉴 | workmux + parallel-code | 模式成熟，代码量小 |
| tmux 管理 | 借鉴 | workmux | Backend ABC 最完善 |
| Agent CLI 调用 | 借鉴 | parallel-code AgentDef | 最清晰的抽象 |
| 状态检测 | 借鉴 | parallel-code | 三状态模型最实用 |
| 自动确认 | 借鉴 | uzi watch | 最简单有效 |
| 合并流程 | 借鉴 | parallel-code + workmux | 锁 + 冲突检测 |
| 编排器 | **自建** | — | 独特需求，无现成方案 |
| Telegram 通知 | **直接复用** | Nanobot | 已有完整实现 |
| 闭环测试 | **自建** | — | 全新领域 |

### Nanobot Elvis Swarm — 零 fork 集成模式

来源：[TECHNICAL-DESIGN.md](../nanobot-elvis-swarm/docs/TECHNICAL-DESIGN.md)

三条设计原则：

1. **复用优先** — Nanobot 已有功能一行不改，只在上层添加集成
2. **零 fork 优先** — 尽量通过 Skill + config + workspace 文件实现
3. **渐进式** — Phase 1 用最少改动跑通核心流程

## 经验教训

### 反锚定偏误

来源：[asr-research-retrospective-20260305.md](../../.claude/memory-bank/learnings/asr-research-retrospective-20260305.md)

> 调研技术选型时横向扫描所有选项，不要被现有方案锚定搜索范围。

反面案例：从现有 paraformer 出发找"离线版"，没横向扫描所有模型，导致路径依赖。

**行动指令：** 调研时先列出所有候选方案类别，再逐一搜索。不要从已知方案出发做增量搜索。

### N=1 不下结论

测新 API 前先查文档确认参数支持。一次成功/失败不代表规律，需要多组验证。

### 验证前提再设计

基于前提设计前先用实际数据验证前提。不要在假设的基础上堆叠设计。

### 可借鉴的外部模式

来源：`~/.claude/memory-bank/research/unattended-claude-automation-20260308.md`

n8n 的错误处理模式可借鉴到 shell 脚本：

| 模式 | 行为 |
|------|------|
| 错误分类路由 | 5xx → 重试；4xx → 人工审查；超时 → 退避重试 |
| 独立错误处理工作流 | 失败通知发 Slack/Email |
| Auto-Retry Engine | 定期扫描失败任务自动重试 |
| Circuit Breaker | 失败超阈值暂停调用，等待后再试 |

## 结论与建议

调研与复用体系已形成完整闭环：

1. **规则驱动** — CLAUDE.md 强制"先调研后编码"，Skill 自动触发
2. **三级决策** — 直接用 / 借鉴 / 自建，每次调研必须输出决策表格
3. **经验积累** — learnings + research 报告永久保留，新项目启动时自动扫描
4. **反锚定** — 复盘提醒横向扫描，不被现有方案锚定

**建议后续改进：**

- claude-mem 的语义搜索目前需要指定 project 参数，跨项目搜索不够便捷
- 复用分析表格可考虑标准化为独立的 JSON/YAML 格式，方便程序化查询
- 高价值开源项目索引 `~/projects/repos/index.md` 需要定期维护
