# 调研: 调研与复用工作流的第一性原理分析

> 从第一性原理出发，分析调研与复用工作流到底在解决什么问题，提炼关键要素，对照业界最佳实践，识别优化空间。

## 调研目标

1. 从第一性原理重新定义核心问题
2. 提炼关键要素，区分本质与偶然
3. 对照外部最佳实践，找到改进方向

## 调研方法

- 内部分析：逐文件提取 CLAUDE.md、Skills、memory-bank 中的 18 条规则
- 外部调研：软件复用理论、AI Coding 工作流、知识管理、开源选型框架
- 来源：Schmidt/Vanderbilt 复用研究、Addy Osmani AI 编码工作流、arXiv Codified Context 论文、McKinsey Build vs Buy、AWS ADR 最佳实践

## 一、第一性原理：到底在解决什么问题

### 根本问题：寻路成本

任何任务都是从 A（现状）到 B（目标）。大模型靠推理找路，本质是在一个巨大的解空间里搜索。但很多时候，**世界上已经有人找到了从 A 到 B 的最优路径**——开源项目、最佳实践、学术论文、社区方案。

不做调研就编码，等于在已有导航地图的情况下闭着眼睛摸索。

这个问题在 AI 编码中尤为严重：

- **LLM 没有记忆** — 每次新会话从零开始，重复犯相同的错误
- **LLM 容易绕圈** — Claude Code、Codex 反复尝试同一类方案，每次绕一个圈子
- **LLM 被训练数据锚定** — 倾向于用训练数据中常见的方案，而非当前最优方案
- **上下文有限** — 无法同时看到所有可选方案，容易局部最优

### 两个本质需求

| # | 需求 | 类比 | 不满足的后果 |
|---|------|------|------------|
| **N1** | **先找路再走路** | 出发前查地图，找到已知最优路径 | 盲目摸索，重复造轮子，反复试错 |
| **N2** | **走过的路记下来** | 把走过的路标注回地图，下次复用 | 同样的弯路走 N 遍，经验无法跨项目传递 |

### 分解为四个核心问题

| # | 问题 | 本质 | 对应 N1/N2 |
|---|------|------|-----------|
| **P1** | **信息不对称** | 不知道世界上已有什么可用的 | N1 |
| **P2** | **决策质量** | 知道了但不知道该选哪个、怎么用 | N1 |
| **P3** | **知识衰减** | 做过的调研和决策会被遗忘 | N2 |
| **P4** | **经验孤岛** | 项目 A 的经验无法自动流入项目 B | N2 |

循环：P1 盲目开发 → P2 选错方案 → 返工 → P3 经验丢失 → P4 下个项目重蹈覆辙 → 回到 P1。

### 现有工作流如何对应

```
P1 信息不对称 ←── research-first + github-search（强制调研，并行子代理）
P2 决策质量   ←── 复用决策矩阵 + ADR（结构化决策）
P3 知识衰减   ←── memory-bank + learnings + claude-mem（经验持久化）
P4 经验孤岛   ←── 全局 memory-bank + 跨项目引用约定（经验流动）
```

## 二、关键要素提炼

从根本问题出发，提炼出 7 个不可或缺的要素：

### 要素 1：任务分解 → 调研主题提取

**是什么**：任何任务都可以提炼出需要调研的要素和主题。调研不是"大项目才做的事"，而是任何任务的默认第一步。

**为什么本质**：如果不主动识别"这个任务的哪些部分可能已有最优解"，就会直接跳进推理模式盲目搜索。提取调研主题是"查地图"的前提——你得先知道要查什么。

**当前实现**：research-first Skill 的触发条件（关键词匹配）+ 复杂度分级。

**改进方向**：从被动触发（匹配关键词）→ 主动提取（任何任务都分析可调研要素）。

### 要素 2：调研信息源矩阵

**是什么**：针对不同类型的任务，知道该去哪里找最优解。

与开发相关的任务，调研信息源包括：

| 信息源 | 适用场景 | 工具 |
|--------|---------|------|
| **开源项目** | 功能实现、工具选型 | gh CLI, GitHub Search |
| **最佳实践** | 架构设计、工作流 | WebSearch, 技术博客 |
| **开源模型** | AI/ML 任务 | HuggingFace, ModelScope |
| **技术社区** | 疑难问题、经验分享 | Stack Overflow, Reddit, Discord |
| **学术论文** | 算法、前沿技术 | arXiv, Google Scholar |
| **市场方案** | SaaS/API、商业工具 | Product Hunt, 竞品分析 |
| **内部经验** | 同类问题的历史解法 | memory-bank, claude-mem, learnings |

**为什么本质**：信息源不同，搜索策略不同。只搜 GitHub 会错过论文里的算法；只搜论文会错过已有的开源实现。全面覆盖才能真正找到最优路径。

### 要素 3：并行调研（默认行为）

**是什么**：调研任务天然可并行。多个调研主题用并行子代理同时搜索，结果汇总后统一决策。

**为什么本质**：串行调研是瓶颈。如果一个任务有 3 个调研主题，串行需要 3 倍时间。并行子代理是"同时查多张地图"。

**当前实现**：research-first Skill 支持启动 Researcher Sub Agent，但没有默认并行。

**改进方向**：调研阶段默认并行——提取出多个调研主题后，每个主题一个子代理。

### 要素 4：复用决策光谱

**是什么**：调研结果不是简单的"用/不用"，而是一个光谱：

| 复用模式 | 说明 | 适用条件 |
|---------|------|---------|
| **直接复用**（含代码） | 整个库/模块/代码片段直接拿来用 | 功能完全匹配，质量可靠 |
| **改造适配** | 基于现有方案做修改和定制 | 功能 70%+ 匹配，需要调整 |
| **借鉴思路，自己重写** | 学习核心算法/架构思想，自己实现 | 思路对但实现不适配 |
| **自建** | 无可借鉴，从零实现 | 全新领域或需求独特 |

**为什么本质**：之前的三级矩阵（直接用/借鉴/自建）缺少"改造适配"这个中间态。实际操作中，fork + 修改是非常常见的复用方式。四级光谱更贴近现实。

### 要素 5：强制触发 + 工具限制

**是什么**：不靠自觉，靠规则自动触发调研。调研阶段硬性禁止写实现代码。

**为什么本质**：Schmidt（Vanderbilt）的复用研究结论 —— 复用失败的根本原因是文化和激励，不是技术。"想起来才调研"等于不调研。强制触发是对抗 AI 急于编码倾向的唯一可靠手段。

**当前实现**：RIPER-5 RESEARCH 模式 + research-first Skill 的 allowed-tools 硬限制。这是现有工作流的最大亮点。

### 要素 6：知识持久化与跨项目流动

**是什么**：调研结果和决策经验必须保存到可检索的位置，并能自动流入其他项目。

**为什么本质**：AI Agent 跨会话天然失忆。没有持久化，每次新对话都从零开始。arXiv Codified Context 论文证实，知识/代码比率达 24.2% 的项目效果最好。

**当前实现**：四层存储（项目 docs/research/ → 全局 memory-bank → learnings/INDEX.md → claude-mem）。

**关键约束**：知识会腐烂。调研报告需要新鲜度标记（`last_verified`），过时信息比没有信息更危险。

### 要素 7：反锚定 + 经验闭环

**是什么**：两个互补机制——

- **反锚定**：防止被已知方案限制搜索范围。调研时先列所有候选类别，再逐一搜索。
- **经验闭环**：偏差时自动复盘，教训写入 learnings，新项目启动时扫描。

**为什么本质**：锚定偏误是系统性的——AI 同样会被 prompt 中的已知方案锚定。没有经验闭环，同样的弯路会反复走。InnerSource 研究（PayPal）显示，建立经验复用文化可减少 56% 开发时间。

## 三、适用场景分析

### 调研主题提取示例

任何任务都可以提取调研主题。关键是问："这个任务的哪些部分，世界上可能已经有最优解？"

| 任务类型 | 可提取的调研主题 | 信息源优先级 |
|---------|----------------|-------------|
| **新项目启动** | 同类项目、技术栈选型、架构模式 | 开源项目 > 最佳实践 > 市场方案 |
| **实现新功能** | 现有库、算法实现、API 设计模式 | 开源项目 > 技术社区 > 学术论文 |
| **技术选型** | 候选方案对比、TCO、社区活跃度 | 最佳实践 > 开源项目 > 市场方案 |
| **Bug 修复** | 类似问题的已知解法、根因模式 | 内部经验 > 技术社区 > 开源 Issues |
| **性能优化** | 瓶颈分析方法、优化算法、基准数据 | 学术论文 > 最佳实践 > 技术社区 |
| **AI/ML 任务** | 预训练模型、数据集、训练技巧 | 开源模型 > 学术论文 > 技术社区 |
| **跨项目集成** | 已有接口、集成模式、兼容性 | 内部经验 > 开源项目 > 最佳实践 |
| **工作流设计** | 同类工作流、自动化工具、编排模式 | 最佳实践 > 开源项目 > 技术社区 |

### 调研深度分级

| 深度 | 条件 | 执行方式 |
|------|------|---------|
| **跳过** | 用户明确说"直接改"；修 typo、调样式；用户已给完整方案 | 直接执行 |
| **轻度** | 单一调研主题，确认用法级别 | 主会话内 ≤3 次工具调用 |
| **中度** | 2-3 个调研主题，需要方案对比 | 并行子代理，每个主题一个 |
| **深度** | 新项目、技术选型、架构设计 | 并行子代理 + 高价值项目深度研究 |

### 不适用的场景

- 紧急修复（生产故障）→ 直接修，事后补调研
- 纯样式/配置调整 → 无需调研
- 用户已给完整方案 → 验证即可，无需发散

## 四、对照业界最佳实践

### 与现有工作流的对照

| 业界实践 | 现有工作流 | 对齐程度 | 说明 |
|---------|-----------|---------|------|
| RPI 三阶段（Research/Plan/Implement） | RIPER-5 五阶段 | ✅ 完全对齐 | RIPER 更细粒度（多了 Innovate 和 Review） |
| CLAUDE.md 作为项目知识载体 | 已全面使用 | ✅ 领先业界 | Anthropic 数据：72.6% 项目在用 |
| ADR 记录技术决策 | 有规则但实际使用较少 | ⚠️ 需加强 | 规则存在，落地不够 |
| OSS 健康度评估清单 | github-search 有筛选标准 | ⚠️ 可增强 | 缺少 TCO 和安全评估维度 |
| Codified Context 三层知识架构 | memory-bank 已有雏形 | ⚠️ 可进化 | 缺少热/温/冷分层和新鲜度标记 |
| Build vs Buy TCO 分析 | 三级矩阵偏定性 | ⚠️ 可增强 | 缺少成本量化维度 |
| InnerSource 跨项目复用 | 跨项目引用约定已有 | ✅ 基本对齐 | 个人项目规模下够用 |
| 经验反馈闭环 | error-retrospective + learnings | ✅ 完整 | 闭环已建立 |

### 三个高价值改进点（来自外部调研）

**改进 1：文档新鲜度标记**

> 规格文档陈旧是 AI 工作流的核心失败模式。Agent 信任过时文档导致错误决策。—— arXiv Codified Context 论文

现有问题：memory-bank 中的调研报告没有 `last_verified` 字段，无法判断信息是否过时。

建议：在 YAML frontmatter 中增加 `last_verified: YYYY-MM-DD`，超过 90 天未验证的标记为 `status: outdated`。

**改进 2：OSS 健康度检查清单**

> 开源复用的五年 TCO 被严重低估。集成开源的实际成本比许可费高 150-200%。—— McKinsey Build vs Buy 分析

现有问题：github-search 的筛选标准偏技术指标（Stars、更新频率），缺少运维和安全维度。

建议在 github-search Skill 中增加评估维度：

| 维度 | 检查项 |
|------|--------|
| 安全 | 已知 CVE？依赖链安全？ |
| 运维成本 | 破坏性更新频率？迁移文档？ |
| 退出成本 | 耦合度？替换难度？ |
| 社区 | 核心维护者数量？Bus factor？ |

**改进 3：轻量 ADR 落地**

> ADR 解决"决策失忆"——正式记录"为什么"而非只记录"是什么"。—— AWS Architecture Blog

现有问题：ADR 规则存在于 CLAUDE.md 和文档规范中，但实际项目中很少创建 ADR。

建议：降低 ADR 创建门槛。技术选型时自动生成 ADR 骨架，不需要手动创建。

## 五、现有工作流的优势

不只看改进，也要识别已经做得好的部分（避免过度优化）：

1. **强制触发是最大亮点** — 多数团队靠自觉调研，这里靠规则 + Skill 工具限制硬约束
2. **三级复用矩阵简洁实用** — 比学术框架轻量，比拍脑袋靠谱
3. **反锚定意识** — 来自实际踩坑的教训，比理论更可执行
4. **四层知识存储** — 项目级 → 全局 → 经验索引 → 对话记录，冗余但可靠
5. **闭环反馈** — error-retrospective 自动触发，不依赖人的自觉

## 结论

### 核心洞察

调研与复用的本质是**寻路问题**：LLM 靠推理在解空间搜索，成本高且容易绕圈；世界上已有大量"已知最优路径"（开源项目、最佳实践、论文、社区方案）。调研就是"出发前查地图"，复用就是"走已知的路"，经验沉淀就是"把走过的路标注回地图"。

### 现有工作流评价

**核心设计正确**，与业界最佳实践（RPI 三阶段、CLAUDE.md 知识载体、经验闭环）高度对齐。强制触发机制是最大亮点，多数团队/工具做不到这一点。

### 最大的改进机会

1. **从被动触发到主动提取** — 任何任务都分析可调研要素，不依赖关键词匹配
2. **调研默认并行** — 多个调研主题同时发出子代理，调研结果汇总后统一决策
3. **四级复用光谱** — 增加"改造适配"模式，更贴近实际操作
4. **信息源矩阵** — 不同任务类型自动匹配不同的搜索策略和信息源

这四个改进的共同方向：让调研更主动、更高效、更精准。不是"要不要调研"的问题，而是"如何用最低成本找到最优路径"。
