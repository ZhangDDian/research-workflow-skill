# 调研工作流引擎

触发后自动声明 `[MODE: RESEARCH]`。

---

## Step 1: 去重检查（L2/L3 必做）

L1 快查跳过此步。L2/L3 调研启动前：

1. 读项目 `docs/research/INDEX.md`（如存在）
2. 读全局 `~/.claude/memory-bank/research/INDEX.md`
3. Grep 关键词匹配已有调研
4. 找到相关 → 提示用户：**复用**（直接引用）/ **刷新**（以旧报告为基础更新）/ **忽略**（全新调研）

---

## Step 2: 分级判断

根据 `contexts/research.md` 的分级规则判断 L1/L2/L3。

---

## Step 3: 场景识别

根据 `contexts/research.md` 的场景→工作流表选择 W1-W6。
组合调研时识别多个工作流，每个分配给一个子代理。

---

## Step 4: 需求澄清 + 受众确认 + 模式选择（按需）

如存在不确定性（功能边界、技术路线、偏好），用 AskUserQuestion 澄清（最多 2-3 个问题）。
L1 和需求已清晰时跳过。

**受众判断**（影响报告语言和深度）：
- `技术` — 可用术语、架构图、代码片段，默认受众
- `业务` — 技术概念需用类比/白话解释，避免术语堆砌
- `高管` — 只要结论和判断，技术细节折叠或省略

从用户指令和上下文推断受众，不确定时问一次。

**搜索模式**（影响子代理执行策略）：
- `递归`（默认） — 假设驱动，每轮搜完评估再定下一轮方向。适合未知领域、需要深挖
- `线性` — 预设关键词，按固定步骤顺序执行。适合已知领域、快速概览、省 token

判断规则：
- 用户说"快速""概览""简单看看" → 线性
- 用户说"深挖""彻底调研""搞清楚" → 递归
- `/research --linear` 或 `/research --recursive` → 显式指定
- 默认 → 递归

---

## Step 4.5: 消化一手信息（用户已提供材料时必做）

当用户随调研请求一起提供了产品资料、截图、文档、对话记录等**一手信息**时，在启动搜索前先完成：

1. **结构化拆解** — 从用户材料中提取关键事实，按维度整理（公司/产品/技术/渠道/定价等）
2. **标注确认度** — 每条信息标注：✅ 已确认（用户明确给出）/ ⚠️ 待验证（推测或模糊）
3. **识别关键假设** — 如果某些推断是后续分析的基础（如"技术路线是 X"），必须列出并在 Step 6 执行中向用户确认
4. **搜索方向对齐** — 基于一手信息确定搜索应该验证什么、补充什么，而非盲目搜

**原则**：用户给的一手信息优先级高于搜索结果。搜索的目的是验证和补充，不是覆盖。

## Step 6: 执行

### L1: 主线程快查

Read/Grep/Glob → 直接回答。无报告、无 PRD。

### L2: 单个子代理

读取工作流模板文件后，启动子代理：

```
# 先读工作流模板
workflow = Read("~/.claude/skills/research/workflows/{W1|W2|W3|W4|W5|W6}.md")

Agent(
  subagent_type: "Researcher",
  description: "调研 {任务简述}",
  prompt: """
    按以下工作流模板执行调研：

    {workflow 内容}

    调研问题：{详细描述}
    搜索模式：{递归/线性}
    受众：{技术/业务/高管}
    验证深度：{🔴/🟡/🟢}
    时效性：{🔥/📅/📚}

    {如果搜索模式=线性，追加：}
    ## 搜索模式覆盖：线性
    不使用 researcher.md 中的递归搜索循环。改为：
    1. 用以下预设关键词各搜 1 次：{主线程根据任务生成 3-5 个关键词}
    2. 从结果中筛选 Top 5 最相关方案
    3. 做对比分析
    4. 给出建议
    5. 写报告
    搜索只在第 1 步做，后面不再搜。不回头修改之前结论。

    {如有一手材料，插入：}
    ## 基准事实（用户已确认，优先级高于搜索结果）
    {结构化拆解后的一手材料}
    搜索目的：验证和补充以上信息，不是覆盖。与基准事实矛盾的搜索结果需标注并保留两方说法。

    工具：按你内置的搜索工具优先级表执行（researcher.md）。
    GitHub 元数据（stars/last commit/语言）已由主线程用 gh CLI 获取，见上方 context。

    质量要求：
    - 每个关键声明附来源 URL（声明-来源配对）
    - 区分"已验证事实"和"推测"，推测显式标注
    - 报告末尾必须有"## 参考来源"章节，按独立来源/利益相关源分类
    - 来自调研对象自身的内容（官博/SEO 对比文/官方账号发布）标注为利益相关源，权重低于第三方

    输出要求：
    1. 筛选后的结果摘要（保留重要细节，去掉噪音和重复）
    2. 写报告到 {存储路径}/YYYY-MM-DD-{主题}.md
    3. 更新对应的 INDEX.md（追加一行）
    4. 返回：摘要 + Top 3 关键来源 URL（附对应声明）+ 文件路径 + 不确定项
  """,
  run_in_background: true
)
```

### L3: 多个并行子代理

识别子领域/工作流组合后，同一个 message 中启动多个 Agent：

```
Agent("Researcher", "按 W1 方案发现: ...", run_in_background: true)
Agent("Researcher", "按 W3 技术评估: ...", run_in_background: true)
Agent("Researcher", "按 W6 文献研究: ...", run_in_background: true)
```

中英双语调研时必须翻译搜索词。

### 子代理失败 Fallback

子代理可能因限流、超时、网络异常等原因失败。处理规则：

1. **不等不重试** — 子代理失败后不重新启动同一个子代理
2. **主线程接管** — 用精简搜索策略（≤5 次工具调用）在主线程完成该子代理的核心任务
3. **降级而非放弃** — 主线程接管时优先覆盖最关键的 2-3 个问题，接受深度不如子代理
4. **标注降级** — 最终报告中标注哪些部分是主线程降级补充的，深度有限

### 假设校验（推断性结论必做）

对技术路线、架构方案等**推断性结论**，在写入报告前必须向用户确认关键假设：

1. 识别本次调研中的核心推断（如"他们用的是 X 技术"）
2. 如果推断是后续分析的基础且无法通过搜索验证，向用户确认
3. 确认后再展开深入分析，避免"推断→基于错误推断继续推断"的链式错误
4. 报告中明确区分 ✅ 已确认 / ⚠️ 用户确认 / ❓ 未验证推测

### 微信公众号源（主线程处理）

当调研涉及微信公众号内容（关键词：公众号、微信文章、mp.weixin.qq.com），**主线程**使用 `wechat-article-extractor` skill 搜索，与 Researcher 子代理**并行**执行。子代理没有 Skill/Bash 工具，无法调用此 skill，所以必须由主线程负责。

全部完成 → 读各子代理摘要 → 汇聚结论。

**汇编质量检查**：汇聚多个子代理结果为综合文档时：
1. 子代理摘要已包含 Top 3 来源 URL，优先使用这些来源构建"参考来源"章节
2. 仅在需要补充细节、验证数据准确性、或来源不足时，才 Read 原始报告文件
3. 综合文档末尾保留"参考来源"章节，按独立来源/利益相关源分类

### 交付前自检（L2/L3 必做）

报告写完后、交付用户前，主线程对报告做一轮检查：

1. **去重** — 同一信息（对比表、风险列表、方案描述）是否出现 >1 次？合并或删除
2. **叙事顺序** — 是否符合"背景→产品→技术→市场→判断"的阅读逻辑？前置依赖是否已交代？
3. **营销 vs 实质** — 调研对象的自我宣传（术语包装、差异化声明）是否被当作事实写入？需降级为"官方声称"或删除
4. **推测标注** — 未验证的推断是否都标了 ⚠️ 或 ❓？
5. **来源偏倚** — 利益相关源（官博/SEO 文）占比是否过高？如 >50%，标注"来源以官方内容为主，独立验证不足"
6. **受众匹配** — 技术术语密度是否匹配目标受众？业务受众的报告里出现未解释的术语 = 不合格

发现问题直接改，不需要再问用户。

## Step 7: PERSIST

子代理已写报告和更新 INDEX.md。主线程确认：
- 文件路径正确
- INDEX.md 已更新（**如不存在则创建**）
- 存储层级正确（项目级 vs 全局级）

**索引自动创建规则**：
- 如果 `docs/research/INDEX.md` 或 `~/.claude/memory-bank/research/INDEX.md` 不存在，主线程在调研完成后**主动创建**
- INDEX.md 格式：`| 日期 | 主题 | 文件路径 | 调研级别 | 关键词 |`
- 确保下次调研时 Step 1 去重检查能命中

## 调研阶段的行为约束

调研阶段专注于信息收集和分析，不进入编码实现：
- 调研阶段不写实现代码——调研完成后切换到研发模式再编码
- 不做最终技术决策——只提供建议和推荐，决策由用户做

## GitHub 搜索规范（原 github-search skill，已合并）

### 搜索命令

```bash
gh search repos "<关键词>" --stars=">100" --sort=stars --limit=10
gh repo view <owner/repo> --json description,stargazersCount,updatedAt
```

### 筛选标准

| 指标 | 阈值 | 原因 |
|------|------|------|
| Stars | > 100 | 基本质量保证 |
| 最近更新 | < 6个月 | 活跃维护 |
| License | MIT/Apache/BSD | 商用友好 |
| Issues 关闭率 | > 50% | 维护者响应 |
| 有 README | 必须 | 基本文档 |
| 依赖数量 | 越少越好 | 轻量优先 |
| TypeScript | 优先 | 类型安全 |
| 测试覆盖 | 有则加分 | 代码质量 |

### 输出格式

对 Top 3 项目输出：项目名 + Stars + 链接、一句话描述、适用场景、潜在问题。

### 采用决策

| 方式 | 条件 | 行动 |
|------|------|------|
| 🟢 直接使用 | 功能完全匹配、维护良好、依赖可接受 | 安装并集成 |
| 🟡 借鉴实现 | 功能部分匹配、或依赖过重 | 阅读核心源码，理解关键逻辑后自己实现 |

在 W1（方案发现）的 SCAN/FILTER 步骤和 W3（技术评估）的 ORIENT 步骤中自动使用。
