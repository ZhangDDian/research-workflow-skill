# [调研模式] 调研上下文

## 核心原则

- 不重复造轮子 — 调研的首要目标
- 先查索引 — 调研前检查已有报告，避免重复劳动
- L2/L3 调研交给子代理执行，主线程保持宏观视角
- 子代理做第一轮筛选，主线程拿到的是精炼后的结果

## 调研分级

| 级别 | 条件 | 执行方式 |
|------|------|----------|
| L1 快查 | 确认用法、查配置、找文件、≤3 次工具调用 | 主线程直接完成，无需 PRD |
| L2 标准 | 技术选型、方案对比、GitHub 搜索、5-15 次工具调用 | 单个 Researcher 子代理 |
| L3 深度 | 多方案对比、跨领域、需网络+代码分析、>15 次工具调用 | 多个并行 Researcher 子代理 |

判断规则：
- 用户说"查一下"/"确认一下" → L1
- 涉及"选型"/"比较"/"GitHub" → L2
- 涉及"完全未知领域"/"多维度对比"/"高风险决策" → L3
- 不确定时默认 L2，执行中发现超出范围可升级到 L3

## 场景→工作流选择

| 用户意图 | 工作流 |
|---------|--------|
| "有没有现成的"/"找轮子"/"竞品" | W1: 方案发现 |
| "用户在抱怨什么"/"痛点" | W2: 用户洞察 |
| "能不能实现"/"技术可行性"/"复用" | W3: 技术评估 |
| "别人怎么做的"/"最佳实践" | W4: 最佳实践 |
| "最新的"/"有什么新技巧" | W5: 知识扩展 |
| "学术上怎么看"/"文献" | W6: 文献研究 |
| 不确定 | 默认 W3（技术评估） |

组合调研：识别 2-3 个相关工作流 → 并行子代理各执行一个 → 主线程汇聚。

## 验证深度（按决策风险叠加在任何工作流上）

| 风险 | 场景 | 要求 |
|------|------|------|
| 🔴 战略级 | 方向选择、架构决策、对外承诺 | ≥3 反驳来源 + 多视角 + 声明溯源 |
| 🟡 选型级 | 技术/工具选型、方案对比 | ≥1 反驳来源 + Top 3 声明溯源 |
| 🟢 信息级 | 查用法、了解现状、获取灵感 | 快速搜反面意见（可选） |

## 时效性路由

| 时效 | 触发 | 搜索约束 |
|------|------|----------|
| 🔥 前沿 | 具体模型名/版本号/刚发布的 API | 限 3 个月内，优先 Tavily/arXiv/HN |
| 📅 近期 | 选型、竞品、趋势 | 限 1 年内，综合多来源 |
| 📚 长效 | 方法论、架构模式、教学法 | 不限时间，优先权威来源 |

## 搜索引擎路由

| 场景 | 默认引擎 | 说明 |
|------|---------|------|
| 全球/英文搜索 | Tavily | 默认首选，实时数据强 |
| 中文/国内搜索 | Baidu MCP | 国内内容覆盖最好 |
| 全球广度补充 | Brave Search MCP | 需要更广泛信息源时启用（如 Tavily 结果不够丰富） |

决策逻辑：
- 常规调研 → Tavily
- 中文关键词/国内平台 → Baidu
- Tavily 结果不足 or 需要更多视角 → 追加 Brave Search
- 多个引擎可并行使用以提高覆盖率

## 工具优先级

1. GitHub: 必须用 gh CLI（gh search repos, gh repo view, gh api search/repositories）
2. 官方文档: Context7 → GitMCP → WebFetch
3. 语义搜索: Exa（理解意图）> Tavily（实时数据）> Brave Search（广度补充）> WebSearch（兜底）
4. 学术: WebSearch "site:arxiv.org" / "site:semanticscholar.org"
5. 中文: Baidu MCP + 知乎/掘金/V2EX
6. 实时热点: HotNews MCP
7. 网页抓取: camoufox（英文反爬）/ Playwright（中文反爬）> WebFetch

## 质量要求

- 关键结论需要多源交叉验证
- 区分"确认的事实"和"推测"
- 每个关键声明附来源 URL（声明-来源配对）
- 不要只看第一页结果，深挖才能找到真正有价值的方案

## 持久化规则

- 项目相关 → 项目 docs/research/ + 更新 INDEX.md
- 跨项目通用 → ~/.claude/memory-bank/research/ + 更新 INDEX.md
- 临时数据 → /tmp/（会话结束可丢弃）
- 报告命名：YYYY-MM-DD-{主题}.md（日期前置，对齐 doc-standard 规范；通用产物如 prd/spec/test-plan 用英文；项目/主题型调研用中文）
- 高价值开源项目克隆到 ~/projects/repos/ 并启动子代理做代码级研究

## 调研→行动衔接

- 调研完成后，明确给出"下一步建议"
- 如需编码 → 提示切换到研发模式
- 如需决策 → 用"选项 + 推荐"格式呈现
- L2/L3 完成后 → 自动生成 PRD 框架草案（推荐方案、核心功能、技术依赖、风险项、待澄清）
