## 二、产品概览

### 做什么

AI 客服智能体——不只回答问题，还试图覆盖主动推销、查订单、改地址、发优惠券等业务动作。覆盖国内私域（微信/千牛/飞鸽/小红书）和海外出海（Shopify/WhatsApp/Messenger）双线渠道。

### 两类接入方式

| 类型 | 适用渠道 | 方式 | 商家门槛 |
|------|---------|------|---------|
| 官方 API 直连 | Shopify、WhatsApp、Messenger、企微、飞书、公众号、邮箱 | OAuth 授权 / SDK | 低（5 分钟） |
| 安卓真机托管 | 淘宝千牛、抖店飞鸽、拼多多、1688、个人微信、小红书私信、视频号 | 商家出安卓机+数据线+电脑 | 中（需硬件） |

### Skill（技能）化设计

每个渠道封装为独立 Skill——独立话术、独立知识库、独立计费、按需启停。商家只为用到的渠道付费。

### 定价

| 版本 | 价格 | 定位 |
|------|------|------|
| Starter | 未公开 | 入门体验 |
| Growth | 980 元/月 | 中小卖家主力版 |
| Pro | 29,800 元/年起，消息约 0.18 元/条 | 中大型商家 |
| Enterprise | 按需报价 | 大客户定制 |

收费逻辑：平台年费（不限坐席）+ 消息条数计费。

### 已知标杆客户

SHOPLINE：企微 AI 客服，实时检索定价体系，按"套餐+权益+优惠"结构化输出。（来源：网易 2026-04；属于公开案例，集成深度和实际使用规模仍需进一步验证）

### GTM 策略

大量"3Chat vs X"对比文营销（知乎/CSDN/搜狐/新浪均为官方账号发布），SEO 意图明显。公开信号看，当前更偏内容获客，尚未看到足够强的第三方口碑传播。

## 三、技术架构

### 1. 整体架构（推断）

以下架构不是官方披露，而是基于产品描述、接入方式和同类 AI 客服/RPA 产品形态的推断。

```
┌─────────────────────────────────────────────────────────────────┐
│                        统一工作台层                               │
│  桌面端（可能兼真机控制宿主）/ Web 后台 / 移动端                  │
└───────────────────────────────┬─────────────────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────┐
│                      AI Agent 调度层                              │
│  Skill Router → Memory/Context → LLM 调用 → 业务系统 API         │
└───────────────────────────────┬─────────────────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────┐
│                      Skill（技能）层                              │
│  每个渠道 = 1 个 Skill（独立话术 / 知识库 / 计费 / 开关）          │
└───────────────────────────────┬─────────────────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────┐
│                      渠道接入层                                   │
│  官方 API 直连（Shopify/WhatsApp/企微等）                         │
│  安卓真机 + USB/自动化控制（千牛/飞鸽/拼多多/微信/小红书等，推断） │
└───────────────────────────────┬─────────────────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────┐
│                      业务系统层                                   │
│  ERP（订单/库存） · WMS（物流） · CRM · 知识库 · 支付（按项目接入）│
└─────────────────────────────────────────────────────────────────┘
```

### 2. 非开放平台怎么接入：安卓真机 + 数据线（高置信推断）

**问题**：千牛、飞鸽、拼多多、个人微信、小红书等平台并不都提供稳定、完整、开放的第三方客服 API。3Chat 怎么做到多渠道接入？

**较可能的方案**：基于产品描述中出现的"安卓机""数据线"等线索，3Chat 对部分非开放渠道大概率采用安卓真机托管/ADB/RPA 类方案：商家准备一台安卓手机，用数据线连接电脑；电脑端软件读取屏幕或控件信息，并执行点击、输入、发送等动作。平台侧看到的是商家的真实设备和官方 App，相比云手机或协议破解更不容易触发设备异常风控。代价是商家需要一台闲置安卓手机 + 一根数据线 + 一台保持开机的电脑。

**为什么不用其他方案？**

| 方案 | 原理 | 可能取舍 |
|------|------|-----------------|
| 云手机 | 服务器上开虚拟安卓机帮商家登录 | 平台检查设备信息，虚拟机的硬件编号/IP 全是陌生的，容易触发风控 |
| 破解协议（如微信 iPad 协议） | 逆向破解 App 和服务器之间的通信格式，直接用程序对话 | 平台更新通信格式后容易失效，封号和合规风险最高 |
| 手机装 RPA App（如影刀） | 在手机上装"自动操作"App，利用无障碍服务权限读屏+点击 | 安卓系统经常杀后台 App，跑着跑着就断了；无障碍权限需反复手动确认 |
| **ADB/真机控制（推断）** | 电脑通过数据线从外部控制手机，用的是商家真机 | 设备指纹更自然、稳定性通常高于手机端 RPA，不一定需要 root |

**各渠道推断接入方式**：

| 渠道 | 接入方式 | 确认度 |
|------|---------|--------|
| 淘宝千牛 / 拼多多 | 商家安卓机 + USB/ADB/RPA 类控制 | ⚠️ 产品描述"要配合安卓机"+需数据线 |
| 抖店飞鸽 / 1688 | 同上（或官方 ISV API） | ⚠️ 推测 |
| 个人微信 | ADB 或 iPad 协议 | ❓ 未验证 |
| 小红书私信 | ADB 或聚光平台官方 API | ⚠️ 聚光已有授权通道 |
| 企微 / 飞书 / WhatsApp / Shopify | 官方 API 直连 | ✅ |

**合规风险**：

| 渠道 | 风险 | 说明 |
|------|------|------|
| 千牛/拼多多 | 🟡 中 | 用商家原设备+官方 App，设备指纹更自然；但自动化操作本身仍可能触发平台规则或行为检测 |
| 个人微信 | 🔴 高 | 个人微信对非人工自动化、外挂、协议或批量控制一贯敏感，封号和合规风险最高 |
| 抖店飞鸽 | 🟡 中 | 抖音有行为检测能力 |
| 企微/WhatsApp/Shopify | 🟢 低 | 官方 API，完全合规 |

### 3. 知识库：URL 爬取自动生成

输入官网/独立站 URL → 自动爬取 → 生成商品知识库（名称/价格/规格/FAQ）→ 向量化存储 → RAG 检索增强生成。支持文档批量上传。这是行业标准技术路线，Intercom/Tidio/来鼓等均采用类似方案。

### 4. 技术演进方向：从真机自动化到 GUI Agent

以豆包手机等端侧 GUI Agent 方向作为参照，下一代方案可能是用视觉大模型"看"屏幕决定下一步操作，不依赖固定控件 ID，App 更新后的适应性更强。

| 维度 | 3Chat 当前（真机自动化，推断） | GUI Agent 方向 |
|------|-------------------|---------------------|
| 感知 | 读控件树 / 截屏 OCR（推断） | 纯视觉大模型理解屏幕 |
| UI 变更容忍 | 中低——App 更新可能中断 | 高——不依赖固定结构 |
| 成本 | 低（商家出硬件） | 高（端侧大模型推理） |
| 延迟 | 快（直接操作） | 慢（截屏→推理→操作） |

3Chat 更现实的中间路线：**真机自动化 + 视觉兜底**——正常走控件树/结构化信息（快），控件树失效时 fallback 到视觉模型识别（慢但更鲁棒）。

```
当前（2025-2026）          中期（2026-2027）           远期（2027+）
真机自动化 + 控件树         真机自动化 + 视觉兜底        全 GUI Agent
快/便宜/脆弱               中等成本/更鲁棒              高成本/最鲁棒
```

### 竞品对比

以下不是严格同类产品对比，而是从商家采购 AI 客服/聚合客服工具时可能进入同一候选池的角度比较。不同产品在目标市场、计费单位和渠道覆盖上并不完全可比。

|------|----------|-------------|------|--------------|-------|----------|
| **定位** | AI Agent + 营销客服 | 聚合聊天 + SCRM | 全渠道 AI 客服 | 跨境电商 AI 客服 | SMB 聊天机器人 | 企业级沟通平台 |

### 市场定位

```
传统客服工具                    3Chat 的位置                      企业级平台
(SaleSmartly/Tidio)            ← 缝隙 →                        (Intercom/Zendesk)
低价 · 规则驱动 · 聊天聚合      中等价位 · AI Agent · 业务深连    高价 · 按座收费 · 重运营
```

3Chat 想填补的是：**需要 AI 执行业务动作（不只答问题），但又难以承担 Intercom/Zendesk 级企业平台成本的中国跨境/私域商家**。

三个结构性机会：
1. **母公司 SaaS 基因** — 新核云有制造业 SaaS 与业务系统集成经验，理论上有利于把 AI 客服从"答知识库"推进到"查单/改单/推品"；但 3Chat 是否已沉淀成熟电商业务动作能力，还需要更多客户案例验证
2. **中国私域碎片化** — 微信/千牛/飞鸽/小红书各有规则，海外工具覆盖不足，3Chat 同时打出海+内贸，渠道组合有差异化
3. **效率→效能升级** — 从"一个面板管消息"升级到"AI 协助销售、跟进、催付"，需求方向成立，但实际效果取决于知识库质量、模型稳定性和人工兜底机制

## 五、结论与判断

### 一句话

**制造业 SaaS 公司（新核云）的第二曲线，疑似用安卓真机/自动化控制补齐非开放平台接入能力，并借母公司业务系统集成经验，切中国跨境/私域商家的 AI 客服需求。**

### 核心竞争力

1. **母公司 SaaS 背景** → 有业务系统集成经验，理论上更容易把 AI 客服做成能查单、改单、推品的业务助手
2. **国内+海外双线渠道** → 微信/千牛/飞鸽/小红书/Shopify/WhatsApp 等覆盖较宽，在国内私域与海外独立站之间形成差异化
3. **Skill 热插拔 + 按消息计费** → 商家按需组合，不限坐席
4. **真机自动化方案（推断）** → 相比云手机/协议号，设备真实性更强、封控风险可能更低，但仍取决于平台规则和操作频率

### 核心风险

1. **规模化瓶颈** — 每个商家一套"手机+线+电脑"物理部署，无法像纯 SaaS 零边际成本扩张
2. **商家体验落差** — 需闲置安卓机+数据线+电脑保持开机，和"扫码 5 秒接入"宣传有差距
3. **脆弱性** — 真机自动化依赖 App UI/控件结构稳定，平台更新可能导致中断；尚未看到 GUI Agent 视觉兜底的公开证据
4. **壁垒薄** — ADB/RPA 本身是成熟技术，真正壁垒在渠道适配稳定性、异常处理、知识库效果、人工兜底和部署运维能力
5. **团队分散** — <50 人同时维护 MES + AI 客服，资源紧张
6. **市场验证早期** — 标杆客户仅 SHOPLINE 一例，Shopify 无高分评价，内容营销主导获客
7. **母公司跨界** — 制造业 MES → 消费级 AI 客服，组织能力和 GTM 经验存疑

### 待验证清单

1. 安卓真机接入是否确认为 ADB，还是自研 RPA/无障碍/混合方案
2. SHOPLINE 案例的上线范围、调用链路和真实使用规模
3. 订单、库存、物流、售后等业务动作是否已经产品化，而非项目制交付
4. 非开放渠道的封号率、掉线率、平台更新后的平均恢复时间
5. Shopify App 评价、续费率、活跃商家数等真实市场牵引数据
